Run a session

  1. Gate — safe for today?
  2. Layer — ball, club, body, equip, psych, strategy?
  3. Visible flaw — choose the pattern before cause hypotheses
  4. Diagnose — pick the version and run the matching proof test card
  5. GP-L — pick the practice rung
  6. Tools — TrackMan / Mach 3 / no-tech / putting / short game / course
  7. Drills — filtered candidates with “why referenced” logic
  8. Build / output — warm-up · blocks · transfer · retest · homework · card

Operating rules

The six rules every Forefront drill card must obey. Built from the QA synthesis.

CB-R · v0.1
  1. One cue. External. Before movement. Never during the rep, never after. Source: Winkelman — Coaching Loop; Wulf & Lewthwaite OPTIMAL 2016.
  2. Body part → fail. Movement effect → pass. "Swing the club through to the target" passes. "Keep your lead arm straight" fails the cue field. Per Wulf & Su 2007 and An & Wulf 2024.
  3. Fade feedback as skill develops. Novice ≥50%, intermediate 33%, advanced bandwidth or self-controlled. Per Barzyk & Gruber 2024 guidance hypothesis.
  4. Distal beats proximal for far-aiming shots. Ball-flight-to-target cues outperform club-path cues for drives and approach. Per Frontiers ecological dynamics 2023.
  5. Constraints first, cues second. If a task manipulation (slope, gate, lie) can produce the correction through self-organization, that is the primary intervention. Per Rob Gray, CLA.
  6. Representativeness matters. Low-fidelity drills carry a transfer note. They are not the primary acquisition vehicle. Per Rob Gray, Representative Learning Design.

Handoff & deployment

How to take this from preview URL to a fixed forefrontops.tech subdomain.

CB-H · ops

Where the data lives

  • data/drills.js — core seed drills with the full schema. data/deepening.js appends the expanded proof-of-concept library, practice-block metadata, and motor-learning stage alignment.
  • data/drills.js also exports the lookup tables: FF_GATES, FF_LAYERS, FF_FAULTS, FF_GPL, FF_TOOLS, FF_GOALS, FF_DURATIONS.
  • assets/style.css — design tokens at the top (--paper, --ink, --teal…). Change once, propagates everywhere.
  • assets/app.js — all UI logic. No build step. No framework. Plain ES6 modules using global namespace.

State model

State is held in a single in-memory FF.state object. Nothing persisted. No localStorage, no cookies, no IndexedDB — per the sandbox rules and Forefront PII policy. Refresh = clean slate.

Deploy now (Perplexity preview)

Parent agent runs deploy_website(project_path="forefront-golf-coach-builder", site_name="Forefront Golf Coach Builder"). Static bundle; no backend.

Move to forefrontops.tech subdomain

  1. Wrap this folder in a Next.js 14 app (App Router). Each top-level view becomes a route: /wizard, /session, /library, /output. The drill array becomes lib/drills.ts exported as a typed const.
  2. Deploy to Vercel under the forefrontops project.
  3. In Vercel project → Domains → add coach.forefrontops.tech (or whichever subdomain). Cloudflare/registrar: add CNAME → cname.vercel-dns.com. TLS auto.
  4. Keep /supabase/migration.sql as a separate, opt-in step — do not auto-run.

Supabase wiring (when ready)

  1. Open supabase/migration.sql. It defines three new tables (forefront_golf_drill_cards, forefront_golf_session_templates, forefront_golf_session_outputs) in their own schema and never touches existing Forefront tables.
  2. Run it manually in the Supabase SQL editor. Verify in information_schema.tables that nothing else changed.
  3. Add the seed step that inserts the static FF_DRILLS rows. RLS policies are stubbed permissive-read, write-restricted — tighten before any client write.

What this PoC does NOT do

  • No DB writes. No mutation. No live data fetch.
  • No client identifiers. No phone numbers. No SMS. No cron. No email.
  • No drill QA enforcement at write time — drill cards are author-curated. The QA rules in Operating Rules are the human-review checklist.
